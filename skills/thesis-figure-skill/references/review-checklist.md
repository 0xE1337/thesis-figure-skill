# 视觉审查与评分清单

> **何时加载**：步骤⑤开始时（渲染完成后、评分前）必须加载本文件。加载后按清单逐项执行审查，全部通过后再打分。

## 评分流程

评分流程（不可跳过任何一步）：
1. **用 Read 工具打开渲染出的 PNG 文件**（TikZ 用 pdftoppm 转的 PNG，draw.io 用 drawio 导出的 PNG）。如果无法生成 PNG，则此步骤阻塞，不可跳过直接评分。
2. **逐项视觉审查**以下清单（每项必须基于你在 PNG 中实际看到的内容，不可基于代码推测）：

## 视觉审查清单（渲染图专用）

**视觉审查清单（渲染图专用，代码检查不替代这些）**：
- [ ] **生命线/连线可见性**：所有连线在背景色上是否清晰可辨？颜色对比度是否足够？（浅色线+浅色背景=不可见，必须加深颜色或加粗线宽）
- [ ] **激活条/标记可见性**：激活条、自调用弧等细节元素在 300dpi 下是否能看到？宽度是否足够？
- [ ] **文字可读性**：所有文字标签（包括数学公式、标注、图例）在 300dpi PNG 中是否清晰可读？最小文字是否过小？
- [ ] **空白区域**：是否存在大面积无内容区域（宽 > 3cm 且高 > 2cm）？某一列/行是否内容过少显得空旷？
- [ ] **元素缺失**：画图指令中列出的每个模块、每条连线、每个标注在渲染图中是否都能找到？（逐条核对）
- [ ] **连线弯折质量**：每条连线是否为连续弧线（单条 `\draw` + `rounded corners`）？弯折点是否有箭头头/尾拼接的痕迹？如果有，说明被错误地拆成了多段 `\draw`，必须合并为单条。
- [ ] **连线穿越文字**：逐条检查每条连线，路径上是否穿过了其他节点的文字或标签？连线标签是否骑在节点文字上方？这是最常见的遗漏问题。
- [ ] **重叠遮挡**：是否有文字被连线、框体或其他文字遮挡？**图例框是否遮挡了公式框、虚线分区框、或其他内容？** 图例必须放在**所有内容元素的完全外部**（zone 虚线框的下方），居中放置，不能和任何分区框边界重叠
- [ ] **整图比例**：纵横比是否合理？是否过窄过长或过宽过扁？放入论文中是否自然？
- [ ] **紧凑度**：各区域之间的间距是否过大？是否存在大面积留白浪费空间？对照参考图——学术配图应紧凑饱满，间距只需"刚好能区分层次"即可，不要过度留白。padding/margin/gap 过大是最常见的问题。
- [ ] **视觉层次**：核心模块是否比辅助模块更醒目？连线粗细/颜色是否能区分主次？

## 设计师视角审查

3. **设计师视角审查**（规则检查之后，交付之前的最终关卡）：

以一个审美严格、经验丰富的学术排版专家身份，**忘掉你写的代码**，只看渲染出的图片。想象这张图即将印在顶会论文里，任何细节瑕疵都会被审稿人注意到。

**第一遍：整体感觉**（5秒钟扫一眼）
- 第一眼看起来**专业**吗？还是"凑合"、"学生作业"的感觉？
- 标题在正确的位置吗？（论文标题应在最顶部，不被子标题挡住）
- 整图比例和谐吗？有没有某个区域特别空或特别挤？
- 有没有某个装饰元素（编号圆、标签、图标）看起来**突兀、不协调**？如果一个元素让你的注意力不自然地被吸引过去，说明它的大小/颜色/位置不对——要么缩小、要么换色、要么去掉
- **编号圆（①②③）是否真正需要**？只有当项目之间有明确的**顺序/优先级关系**时才加编号。如果是并列的、地位相等的项目（如多个研究子课题），编号反而暗示了不存在的顺序，应该去掉。**默认不加编号**，除非顺序信息是必要的
- **装饰元素的使用是否前后一致**？如果某个区域有编号圆①②③但同类的其他区域没有，读者会困惑——要么全部加，要么全部不加。不一致的装饰比没有装饰更差

**第二遍：逐元素审查**（放大到 100% 逐个看）
- 每个文字标签都能**不费力地阅读**吗？有没有被箭头线穿过、被白色方块截断、被其他元素遮挡？
- 节点框下方有没有"外挂标注"被连线穿过？如果有，这些标注应该合并到框内（`\\ \scriptsize 说明`），不要外挂。
- 连线标签是优雅地"浮在线旁边"还是粗暴地"骑在线上用白底遮住箭头"？后者必须修复。
- 竖排文字是真的竖排（每个字从上到下排列）还是需要歪头才能阅读？
- 框内文字是否太小？如果框有足够空间，文字应该大一些（fontSize ≥ 11）。
- 有没有元素之间差了几个像素没对齐？（如圆柱体的连线没接到侧栏卡片）
- 有没有太短的箭头（只有头没有线段）？这些箭头是多余的，应删除。
- **区域内部有没有不必要的空白**？——每个分组框、虚线框、卡片区域内部的元素应该紧凑排列。如果一个框内只有3个小卡片但框很大，要么缩小框、要么增加内容、要么调整卡片大小填满空间。**内部空白和外部留白不同**——外部留白用于分隔层级，内部空白是浪费。
- **同行同类元素是否等宽等高**？——同一行的多个卡片/框必须宽度一致、高度一致。这是最基本的排版规范，宽窄不一看起来非常不专业。检查每一行并排的元素。

**第三遍：连线路径**（沿每条线从起点到终点走一遍）
- 每条连线是一条连续的弧线吗？还是在弯折处有两段线拼接的痕迹？
- 连线有没有穿过任何框体或文字？
- 连线的弯折是圆角还是死板直角？
- **这条线的路径是最优的吗？**——在脑中想象 2-3 条替代路径，有没有更短、更少穿越、更自然的走法？特别是回调/反馈线，它们容易走出不必要的弯路或穿越不该穿的区域。如果你能想到更好的路径，就必须改。

任何一个问题都必须修复后才能进入打分环节。**这一步的标准是"挑不出毛病"，而非"大致没问题"。**

**迭代修改中的审美守恒原则**：修复一个问题时，**不能让其他方面变差**。具体做法：
- 每次迭代修改后，不仅要检查修复的问题是否解决了，还要检查修改是否**引入了新问题**或**破坏了已有的好效果**
- 特别注意：对称性、平行关系、间距均匀度——这些整体美感特征容易在局部修改中被无意破坏
- 如果发现修改后某个方面退步了（比如之前两条线是平行的，现在不平行了），必须回退或调整，**不允许以修复一个 bug 为代价引入另一个审美问题**

## 像人一样思考的核心方法

**像人一样思考的核心方法**：

审查时不要逐条对照规则清单——而是**想象自己是第一次看到这张图的读者**。你的眼睛会自然地扫过整张图，注意力会被不自然的地方吸引。任何让你"觉得怪怪的"的地方都是问题，即使你说不出违反了哪条规则。

**审美直觉的核心原则**（不是规则清单，是思考方式）：

1. **"这里怪怪的"就是问题**：不需要找到违反了哪条具体规则。如果你盯着某个地方觉得不舒服——两条线差一点平行但没完全平行、两个框差一点对齐但没完全对齐、一个标签差一点就不和曲线重叠但还是碰到了——这些"差一点"就是问题。人的眼睛对"几乎但没完全"比"完全没有"更敏感。

2. **层次分明感**：好的图有明确的视觉层次——文字在最上面，框体在中间，箭头线在最下面。如果箭头线盖住了文字，层次就乱了。问自己：这张图的前景、中景、背景分别是什么？它们的层次对吗？

3. **呼吸感**：每个元素周围都需要一圈"空气"——标签不能紧贴曲线、框不能紧贴框、文字不能紧贴边界。但空气也不能太多——元素之间如果离得太远，图就显得空旷松散。**紧凑但不拥挤**是标准。

4. **一致性直觉**：人眼会自动寻找规律。如果两条线看起来应该平行（比如上传和下载），那它们就必须严格平行。如果三个框看起来应该等宽，那它们就必须严格等宽。**"看起来应该一样"的东西如果不一样，比"本来就不一样"的东西更刺眼。** 一致性的具体体现：
   - **弯折风格全图统一**：如果全图用圆角弯折（`rounded corners`），那**每一条**连线都必须是圆角，不能有一条突然变成尖角——那条线会像刺一样扎眼
   - **想要水平箭头→先对齐 y**：如果你希望 A→B 是一条水平直线，A 和 B 的 y 坐标必须相同。画图指令阶段就要规划好同一行模块的 y 坐标一致，不要画完才发现箭头是斜的
   - **箭头入口点适中**：箭头不要指向目标框的最边缘（`.west` 的最左端或 `.east` 的最右端），特别是当多条线从不同方向进入时——入口点靠近中心区域更自然。用 `[yshift=±5pt]node.north` 比 `.north west` 好看

5. **线条的优雅感——"能直就直"**：好的连线像水流——弯折干净利落（严格 90 度圆角），路径选择走最短最自然的那条。**核心原则是"能直就直"**：如果源和目标在同一 x 或同一 y 上，直接一条直线连过去，不要画 L 型弯折。AI 容易犯的错误是"默认画 L 型"——即使源和目标完全对齐也加一个不必要的弯折。**每次弯折都要有理由**（绕开障碍物、避免穿越文字），没有理由的弯折就是多余的。此外：
   - **弯折后的线段不能太短**：弯折点和箭头尖之间至少 0.8cm 的直线段。如果弯折点就是箭头——看起来像箭头断在了弯角处，非常丑。解决方法：提前弯折，让最后一段线段足够长
   - **平行段必须严格平行**：如果路径中有两段应该平行的线段（水平段-垂直段-水平段的 U 型），两个水平段的 y 坐标必须完全相同。差 1-2px 就会产生微斜线段，人眼对这种"几乎平行但没完全平行"极其敏感
   - **树状扇出的垂直段必须严格垂直**：分支起点的 x 坐标必须精确等于目标节点的 x 坐标——不能手动估算。用 TikZ 坐标提取：`let \p1=(target.north) in (\x1, 横线y) -- (target.north)`。手动写一个近似 x 值是微斜线的头号元凶
   - **树状扇出只向下不向上**：向下扇出（父在上、子在下）像树枝分叉，自然好看。向上扇出（源在下、目标在上）像倒立的树，弯折处看起来"拐过去就戳进框底"，很丑。向上一对多改用独立 L 型连线

6. **同类线条的出发点选择**：之前说"多条线从同一节点出发用不同锚点"——但这不是绝对的。当两条**同类型**的线（都是实线、同色、去往相似方向）从同一节点出发时，**共享一个出发点反而更干净**——它传达的是"同一批数据分发"。只有当线的**类型不同**（实线 vs 虚线）或**方向差异大**时才分开。**判断标准**：共享时看起来是"有意为之"还是"挤在一起"？前者共享，后者分开。不要机械地执行"必须分开"的规则。

7. **箭头的方向感**（最常被忽视的问题）：每条箭头必须在视觉上"流向"它逻辑上要去的方向。人看图时会直觉地感受到"这个箭头方向不对"——即使箭头最终连到了正确的目标。**核心问题是出发锚点和到达锚点的选择**：
   - 如果目标在源的**上方**，箭头应从 `.north` 出发（向上走），不能从 `.east` 绕一圈再向上——那看起来像是箭头先往右走再折回来，视觉上"逆流"了
   - 如果目标在源的**右边**，箭头应从 `.east` 出发，到达目标的 `.west`——不能从源的 `.west` 出发绕大弯到目标的 `.east`，那看起来完全是反的
   - **验证方法**：画完每条箭头后，问自己——"如果我是读者，我第一眼会觉得这条线在往哪个方向走？"如果第一眼的方向感和逻辑上的数据流方向**矛盾**，就说明锚点选错了。人对这种"看起来反了"极其敏感

8. **做减法的勇气**：编号圆、装饰标签、白色背景方块、大括号维度标注——每个元素都要问"它在传达什么信息？"如果答案是"纯装饰、没有信息量"或者"加了反而看起来怪"，就去掉。**大括号标注向量维度**：如果大括号方向不对（开口朝错方向）或者显得突兀，直接去掉改用文字标注（如 `$d_k$`）。少一个不需要的元素，比多一个有问题的元素更好。

9. **规则不是死的，美感是活的**：所有规则都是"通常情况下的好实践"，不是"任何情况都必须执行"。当严格执行某条规则导致了比不执行更丑的结果（比如强制分开出发点导致路径扭曲、强制加装饰导致视觉混乱），就应该违反这条规则。**最终评判标准只有一个：看起来好不好。** 规则帮你避免常见错误，但审美直觉是最终裁判。

## 具体检查方向

**具体检查方向**（审美直觉的具体应用）：

1. **连线悬空**：线的两端都连在节点上吗？有没有"连到空气"的线？
2. **文字方向**：所有文字都能自然阅读吗？竖排从上到下？
3. **遮挡层次**：文字 > 框体 > 箭头线。如果箭头在文字上方，层次反了。**所有文字标签都可能被连线盖住**——不仅是 zone 标题，还有阶段编号胶囊（如"④ Decision Engine"）、分区标签、图例标题等。解决方法：所有独立文字标签用 `\node` 绘制，放在所有 `\draw` 连线之后（TikZ 后画的在上层）。渲染后逐个文字标签检查：有没有连线穿过？
4. **微重叠**：两个元素之间有没有 1-2px 的轻微重叠？人眼对"差一点碰到"极其敏感——要么完全不碰（留足空气），要么明确重叠（如标签在线上），"差一点碰到"最难看。**垂直排列的框尤其容易重叠**——上面的框因为文字行数多、inner sep 大，实际高度超出预期，底边侵入下面框的空间。解决方法：用 `below=0.8cm of` 而不是 `0.3cm`，宁可间距大一点也不要重叠。渲染后逐对检查相邻框：两个框的边框线之间是否有清晰的间隙？如果贴在一起或重叠了，加大间距。
5. **居中对齐**：标题居中了吗？底部元素居中了吗？**嵌入可视化在框内水平居中了吗？**——scope shift 的 x 坐标必须让可视化（含左侧标签和右侧数值）整体居中于容器框。最常见的偏移：横条图整体偏右（因为左侧类别名占了空间但 scope x 没有补偿）。渲染后逐个可视化框检查：左右留白是否大致相等？
6. **字图比例**：图大字就应该大，图小字就可以小。
7. **时序图骨架**：生命线从上到下连续可见，是时序图的骨架。
8. **箭头方向感逐条审查**（最重要的审查项之一）：**沿着每条箭头，从起点到终点用眼睛"走"一遍**。问自己：这条箭头第一眼看起来在往哪个方向走？这个方向和数据/控制流的逻辑方向一致吗？如果一条"数据从下到上流"的箭头，视觉上看起来像是先往右/左绕了一圈——那就是锚点选错了。**特别注意进入锚点的选择**：如果源在目标的左下方，箭头应该先向上走（从源 `.north`），到达目标的 y 高度后水平右行，从目标 `.north` 进入——而不是直接水平进入目标 `.west`。水平进入意味着"从左边来"，但实际数据是"从下面来"。**进入方向必须反映源的位置**：源在下方就从 `.north` 进入，源在左方才从 `.west` 进入。
9. **标注必须放在空旷区域，不能挤在数据密集区**：几何/数学示意图中，标注（margin 箭头、公式标签、"支持向量"等文字）不能放在数据点、线条密集的区域——挤在一起读者什么都看不清。**正确做法**：找到图中最空旷的角落或边缘放标注，用细指引线（leader line）连接标注到它标注的元素。如果标注附近 1cm 内有 2 个以上其他元素（数据点、线、标签），就必须把标注移到更空的地方。渲染后检查：每个标注周围是否有足够"空气"？标注是否和其他标签重叠或紧贴？
10. **虚线/反馈线逐段审查**：虚线（反馈、回调）是最容易出问题的线——因为它通常要走很远的路径。**必须逐段检查**：这条虚线经过的每一段路径上有没有文字？有没有穿过框体？如果有——哪怕只穿过一个标签——就必须修改路径绕开。虚线穿过文字是最明显的排版问题，人一眼就看到，但 AI 容易忽略因为虚线在代码中只是一行 `\draw`。
11. **装饰元素方向正确性**：大括号、箭头、弧线等装饰元素有固有的方向性。大括号 `{` 开口朝右标注右侧内容，`}` 开口朝左标注左侧内容。如果方向搞反了（大括号开口背对内容），比没有还难看——直接去掉改用文字标注更好。**TikZ 大括号方向速查表**（路径方向 + mirror 组合）：

| 路径方向 | 无 mirror | 有 mirror |
|---------|----------|----------|
| 向下 (y_top→y_bottom) | 开口朝**左** | 开口朝**右** |
| 向上 (y_bottom→y_top) | 开口朝**右** | 开口朝**左** |
| 向右 (x_left→x_right) | 开口朝**上** | 开口朝**下** |
| 向左 (x_right→x_left) | 开口朝**下** | 开口朝**上** |

**生成大括号后必须检查**：大括号开口是否朝向它标注的内容？左侧标注右边内容→开口朝右，右侧标注左边内容→开口朝左。查表确认 path direction + mirror 组合是否正确。
12. **draw.io 背景层必须在最底层**：区域背景色（layer backgrounds）必须在所有内容元素（框、文字、箭头）的**下方**。如果背景色跑到了箭头或文字上方，说明 z-order 错误。在 XML 中确保背景 mxCell 在所有内容 mxCell 之前定义。
13. **"能直就直"逐条检查**：逐条检查每条连线——源和目标是否在同一 x 或同一 y 上？如果是，这条线应该是一条**直线**，不该有弯折。如果有不必要的弯折，说明路径规划时没检查对齐关系。**draw.io 中尤其常见**——明明两个框垂直对齐，箭头却弯了一下。
14. **弯折后线段长度检查**：弯折点到目标框之间距离 ≥ **1.5cm**。`rounded corners` 弧线过渡会"吃掉"约 0.5-0.7cm，所以 0.8cm 不够——弯折点距目标 0.8cm 时圆角后只剩 0.1cm 直线段，箭头"刚拐过来就到了"。1.5cm 确保圆角后仍有 ≥ 0.8cm 可见直线段。
15. **标签逐个检查放置位置**：Yes/No、协议名、连线标注——每个标签都要单独检查它和周围所有元素的关系。标签是否和线重叠？和框重叠？和其他标签重叠？如果重叠了，周围有没有空白可以挪过去？**有空白却还重叠放是最不应该的错误**——说明画图时根本没看周围环境。
16. **箭头必须"够到"目标**：自调用弧的终点必须落在激活条或节点的边框上，不能指向空气。如果目标元素（如激活条）太短或太窄导致箭头够不到，**扩大目标元素**而不是缩短箭头。
17. **draw.io 内容居中**：容器框内的内容元素必须水平和垂直居中。如果内容偏左偏右偏上偏下，调整内容位置或容器大小。
18. **外挂标注逐框检查**：渲染后逐个框检查——这个框的下面/上面有没有独立的小标注（如"特征流""调度状态"）？如果有，它**一定会**被连线穿过——因为框的下方/上方就是连线经过的区域。必须合并到框内作为第二行文字。这是最顽固的问题——规则写了很多遍但每次还是有人犯。
19. **虚线绝不共享路径段**：两条虚线如果在同一段路径上重叠（共享同一 rail 的同一段 x 或 y），读者完全分不清哪条是哪条——实线重叠还能靠连续性辨认，虚线重叠就是一团乱麻。**解决方法**：（a）两条虚线用不同侧的 rail（一条走左侧、一条走右侧）；（b）如果必须同侧，两条 rail 间距 ≥ 1.5cm；（c）更好的做法是**重新审视路径设计**——如果一条虚线可以走更短的直连路径而不需要绕 rail，就不要绕。反馈虚线和 zone 边框虚线也不能在同一区域交叉，rail 必须在 zone 边框外侧 ≥ 1.0cm。
20. **微斜线是最顽固的 bug——必须在代码层面验证，渲染后目视不可靠**：这个问题从 batch7 到 batch18 每轮都出现，说明"看 PNG 检查"根本发现不了微斜线。**唯一可靠的验证方法是审查代码**：逐条 `\draw` 路径，检查每一对相邻坐标点是否共享 x 或 y。例如 `(2.0, 8.0) -- (2.5, 6.5)` 中 x 从 2.0 变到 2.5、y 从 8.0 变到 6.5——两个坐标都变了，这就是斜线。必须改为 `(2.0, 8.0) -- (2.0, 6.5)` 或 `(2.0, 8.0) -- (2.5, 8.0) -- (2.5, 6.5)`。**代码提交前的强制检查**：对每条包含 `rounded corners` 的 `\draw`，列出路径中所有坐标点，逐对验证共享 x 或 y。不通过就不提交。
21. **箭头方向逐条语义验证**：不只看箭头"连到了正确的目标"，还要验证**箭头尖的方向是否符合数据流语义**。例如"PS 广播模型到 Worker"——箭头尖必须指向 Worker（数据流入 Worker），不能反过来。常见错误：画回路线（PS→Worker→PS）时，下行箭头（PS→Worker）画反成了上行（Worker→PS）。**验证方法**：读出每条箭头的语义——"A 把 X 发给 B"，检查箭头尖是否在 B 端。
22. **时序图消息箭头必须水平**：每条消息线的两端必须在同一 y 坐标上。斜线消息在 UML 时序图中是错误的（暗示传输延迟）。渲染后放大检查每条消息是否严格水平。
23. **zone 标题与内容框不重叠**：zone 的角落标题和 zone 内的内容框（公式框、注释框）如果靠近同一个角落就会重叠。渲染后检查每个 zone 标题附近有没有框体。
24. **分区标签有空间放**：对比图/架构图中的分区标签（"控制平面""数据平面"）如果和节点文字重叠，说明 zone 框太小。加大 zone 高度让标签有自己的空间——标签不是"挤进去"的，而是"本来就有位置放"。
25. **多源→多目标连线禁止交叉斜线**：当多个源节点（如"结果A""结果B"）需要连接到多个共享目标（如底部评估指标），**绝对禁止斜线交叉**——斜线交叉是最丑的连线模式。**首选树状汇聚-扇出**：多个源通过汇聚树（rounded corners）合流到中心树干，树干再通过扇出树分叉到各目标——整体形成沙漏形，流畅自然。如果中间有标题文字，用 `fill=white` 覆盖树干（标题画在连线之后）。**不要用点状 bus**（水平线+圆点连接点）——点状连接显得机械、别扭，缺乏流动感。**审查时看底部连线区域：有没有线交叉成 X 形？有就必须重新路由。**
26. **弯折后箭头必须有足够直线段——特别是树状扇出**：`rounded corners` 弧线过渡会"吃掉"0.5-0.7cm 线段。弯折点到目标框 ≥ 1.5cm。**树状扇出的垂直分支同样适用**：横线到目标框的垂直距离也必须 ≥ 1.5cm——如果横线紧贴目标框（只有 0.5cm），分支线"刚从横线拐过来就是箭头"，极其难看。**计算横线 y 坐标时**：`横线y = 目标框底部y - 1.5cm`（向下扇出）或 `横线y = 目标框顶部y + 1.5cm`（向上扇出）。渲染后放大检查**每一条树状分支**：从横线到箭头之间是否有清晰可见的直线段？
27. **禁止无信息量的装饰线条**：不要在形状内部添加"装饰性曲线"（如椭圆内部画一条小弧线暗示曲线形状）——这些线在渲染后看起来像头发丝或污渍，读者不知道它们代表什么。**每个视觉元素都必须传达明确信息**——如果删掉它不影响读者理解，就应该删掉。
28. **Hub-spoke 辅助连线要克制**：hub-spoke 图的核心是"中心→辐射"的清晰结构。spoke 之间的辅助连线（闭环、知识库支持线等）**最多 2-3 条**——超过这个数量整张图就变成了蜘蛛网。如果逻辑上确实有很多 spoke 间关系，只保留最核心的闭环路径，其余用文字说明或注释框补充，不要画线。
29. **多箭头入同一节点必须分散锚点**：当 3 条以上箭头进入同一个节点时，不要全用 `yshift` 微调同一个 `.west` 或 `.north`——挤在一起会产生扭曲变形、非90度弯折。**正确做法**：用不同锚点（`.north`、`.west`、`.north west` 等），让每条线从不同方向自然进入。如果必须同侧进入，锚点间距 ≥ 0.5cm，且每条线有独立的弯折路径。渲染后检查：节点附近有没有线挤成一团或弯成非90度的扭曲？
30. **图例框放在最空旷的角落**：几何/散点类图中，图例框必须放在离数据最远的空白角落，**绝对不能和数据点、投影线、坐标轴重叠**。渲染后检查：图例框区域内有没有被遮挡的视觉元素？
31. **draw.io 竖排文字从上往下读**：左侧时间标注（如"3个月"）用竖排时，必须**从上到下自然阅读**，不能倒着读。渲染后：竖排文字是否需要歪头？需要歪头就是方向反了。
32. **draw.io 路线图紧凑排列**：阶段间连接箭头长度控制在 **20-40px**，不要留大段空白。整个路线图应该紧密衔接，阶段之间的垂直间距不超过阶段标题栏高度的 1.5 倍。渲染后如果感觉"散"——阶段之间有大面积空白——就缩短箭头和间距。
33. **嵌入内容不能溢出容器框**（生成和审查都必须执行的最高优先级检查）：

**生成阶段（写代码时预防）**：每画一个含嵌入可视化的框，**必须先算再画**——
1. 列出框内所有内容的高度：标题(~0.5cm) + padding(0.3cm) + 可视化高度 + x轴标签(~0.4cm) + 底部padding(0.5cm)
2. 框高度 = 上述总和，**不能"差不多"要精确到 0.1cm**
3. 最容易遗漏的：**柱状图/条形图底部的 x 轴文字标签**（如"本系统""BLEU-4"），这些标签在 scope 内 y=-0.25 左右，必须算入总高度
4. 用注释写明验证：`% 框底 y=9.5, 标签底 y=10.07, padding=0.57cm ✓`

**审查阶段（渲染后逐框检查）**：逐个含嵌入可视化的框，放大到 200% 看——
- 可视化的 x 轴标签文字是否完全在框线内部？（最常溢出的位置）
- 热力图/矩阵的标签是否在框内？
- 框四边到最近内容的 padding 是否 ≥ 0.3cm？
- 如果发现贴边或溢出：加大框高度/宽度 0.5cm，不要缩小内容

34. **箭头路径中间点不能超越目标（最阴险的方向 bug）**：`\draw[arrow] (A) -- (mid) -- (B)` 中，如果 mid 的 y 坐标**超过**了 B 的 y 坐标（如 mid 在 y=5.65，B 在 y=5.4），最后一段路径方向反转，箭头尖指向反方向——箭头连到了正确位置但头朝反了。**修复**：同轴路径（A、mid、B 的 x 相同）删掉多余中间点直连；不同轴时确保每个中间点的坐标值严格在前一点和目标之间。生成代码后逐条 `\draw` 检查：每对相邻点的 y(或 x) 是否单调递增/递减？不单调就是潜在方向反转。
35. **箭头头大小必须匹配箭身长度**：`Stealth[scale=X]` 的头部占约 0.15×X cm。在 0.5cm 短箭头上用 `scale=1.5` 几乎没有可见箭身——看起来是实心三角形而非箭头。**尺度指南**：箭身 ≥ 2cm → scale ≤ 1.2；1-2cm → scale ≤ 0.9；0.5-1cm → scale ≤ 0.7；< 0.5cm → 箭头太短，必须加大间距。全局定义的 `phase_arrow` 等样式如果 scale 很大，在短箭头场合必须用内联覆盖：`\draw[-{Stealth[scale=0.8]}, very thick]`。
36. **区域边框紧凑 + 跨区箭头从边界出发 + 标签居中**：三条相关布局规则——(a) 背景区域矩形（zone/region）到最外层内容的 padding ≤ 1cm，空带浪费空间；(b) 箭头从一个 zone 到另一个 zone 时，从**源 zone 的边界**出发，不从 zone 中心出发——从中心出发意味着箭头切过 zone 内部，视觉上不对；(c) 输出标签（如"Encoder Output"）、区域副标题等必须水平居中于其逻辑父元素，x 坐标与父元素中心一致。
37. **小型汇合节点（⊕、⊗）两侧需要间隙**：直径 ~0.5cm 的圆形节点夹在两个大框之间时，上下各需 ≥ 0.4cm 间隙才能看到箭身。验证：`上方框底 + 0.4 ≤ ⊕顶部` 且 `⊕底部 + 0.4 ≤ 下方框顶`。不满足时拉开框距，不要缩小⊕。
38. **箭头必须刚好止于目标框边界，不能刺入内部**：硬编码箭头终点坐标时，必须精确匹配目标框的实际边界坐标。如果框顶 y=3.0，箭头终点必须是 y=3.0 而非 y=2.6——差 0.4cm 就刺入了框内，箭头尖在框的标题/内容上方，极其难看。**验证方法**：每条指向框的箭头，检查终点坐标是否等于目标框的 `rectangle` 边界值。
39. **坐标轴标签与刻度值防重叠**：嵌入图表中最常见的文字重叠有两种——(a) x 轴标题（如"Iterations (×10³)"）和 x 轴刻度值（如"400"）重叠：给标题加 `yshift=-10pt`；(b) y 轴旋转标题（如"Loss"）和 y 轴刻度值（如"0.5"）重叠：给标题加 `xshift=-15pt`。生成嵌入图表后必须逐轴检查：标题文字和最近的刻度值之间是否有 ≥ 0.3cm 间隙？
40. **公式框宽度必须留余量**：含数学公式（尤其 `\mathbb{E}`、`\left[\right]`、`\|`）的框，宽度必须比公式渲染宽度多 ≥ 1cm（两侧各 0.5cm padding）。公式宽度难以在代码中精确预估——**宁可框宽一点也不要公式贴边**。渲染后逐个公式框检查：公式最宽行到左右框边距 ≥ 0.3cm？
41. **zone 边框必须完全包含内部内容**：zone/region 的背景矩形坐标必须覆盖其内所有文字标签和框体。最常见的遗漏：(a) zone 虚线切过边缘附近的标签（如 "Create Structure"）；(b) zone 背景没有覆盖到下方面板。渲染后逐个 zone 检查：有没有内容露在背景色外面？有没有虚线穿过文字？
42. **Stage/区域标题与首行内容间距 ≥ 0.4cm**：标题框占 ~0.4cm 高度，标题 y 和内容 y 之差如果 < 0.8cm，可见间距就不足。渲染后检查每个标题下方：标题底部到第一行内容顶部是否有清晰的空白带？
43. **同层相邻框不能 x 方向重叠**：同一行/层的多个框如果 y 范围相似，必须检查 x 范围有无交集。最常见：左侧信息框右边界超过右侧可视化框左边界。间距 ≥ 0.3cm。
44. **迷你可视化标注放在元素外侧**：小型图表内的数值标注（如注意力权重 .35）不要用插值定位（落在元素边缘上），改用 anchor 定位放在元素正外侧。渲染后放大检查：标注文字有没有和它标注的圆圈/方块重叠？

## 多 agent 分工评审

**多 agent 分工评审**（推荐做法）：生成图片后，启动多个子 agent 并行评审，每个专注一个维度：
- **排版审查 agent**：检查对齐、居中、间距均匀、元素是否偏移
- **连线审查 agent**：逐条检查连线路径——是否穿越文字/框体、弯折是否连续、箭头是否碰撞（两条线在同一区域交叉或平行太近）
- **文字审查 agent**：检查所有文字可读性——字号、方向（竖排从上到下？）、是否被遮挡、标签位置
- **配色审查 agent**：检查颜色和谐度、对比度、同类元素是否一致

每个 agent 独立输出问题清单，汇总后逐项修复。这比单人审查严格得多——单人容易"看了一眼觉得还行就过了"，分工后每个维度都被专人盯着。

## 六维度评分

4. **基于视觉审查结果打分**（六维度，各 1-5 分）：

| 维度 | 评分依据（必须基于 PNG 所见） |
|------|--------------------------|
| 完整性 | PNG 中能看到画图指令中的所有模块和连线 |
| 准确性 | 文字内容正确，模块归属关系与指令一致 |
| 布局合理 | 无大面积空白，间距均匀，比例协调 |
| 连线清晰 | 每条连线清晰可辨，走向自然，无穿越遮挡 |
| 配色统一 | 颜色对比度足够，同类元素颜色一致 |
| 文字可读 | 最小文字在 300dpi 下可读，无重叠遮挡 |

**评分规则**：
- **总分 = 30 且视觉审查清单全部通过** → 交付
- **总分 < 30 或任一审查项未通过** → 继续迭代（定位问题 → 修改代码 → 重新编译 → 重新查看 PNG → 重新评分）
- **总分 < 25** → 重大问题，回到步骤①重新设计

## 常见"代码正确但渲染不佳"的陷阱

**常见"代码正确但渲染不佳"的陷阱**（agent 自评满分但实际不合格的典型原因）：
- 激活条宽度 0.3cm 在代码中看起来合理，但渲染后在 300dpi PNG 中几乎不可见 → 加宽到 0.4-0.5cm
- 生命线用 `color=xxx!60`（60% 透明度），在浅色阶段背景上几乎消失 → 改为 `!80` 或更深
- combo 框用浅灰虚线，在浅色阶段背景上看不到 → 加深颜色或加粗线宽
- 连线用浅橙/浅黄色在白色背景上对比度不足 → 用更深的颜色
- 右侧参与方/模块很少有交互，导致该列空旷 → 增加注释框或重新规划参与方顺序
- 整图纵横比 1:2.5 以上，嵌入论文时会被压缩到很小 → 控制纵向长度或增加横向宽度
- 各区域间 padding/margin/gap 过大，整图看起来稀疏空旷 → 学术配图应紧凑饱满，缩减间距到"刚好能区分层次"即可
- 跨层连线被拆成多段 `\draw`，弯折点出现箭头头/尾拼接痕迹 → 改为单条 `\draw` + 路径点 + `rounded corners=6pt`
- 连线标签骑在其他节点文字上方，造成视觉混乱 → 标签只放在连线经过空白区域的段上
- 连线穿过分组框或节点文字 → 重新规划路径，走 gap 区域或框外侧
